## 1. 项目概述

### 1.1 项目背景
ACM竞赛教学与知识传播过程中，传统的幻灯片制作方式存在明显痛点：
- **排版效率低**：PPT制作需要大量时间调整格式，与算法内容的逻辑表达脱节；
- **演示与代码分离**：算法代码与动态演示难以同步，影响教学直观性；
- **内容复用困难**：笔记、代码、演示素材分散，难以形成体系化知识库。

### 1.2 项目目标
打造一款**面向算法教学与展示的轻量级开发工具**，核心理念为“内容至上，演示即代码”。让使用者专注于算法知识的输出，工具自动完成排版与演示生成，实现高效、直观的教学体验。同时，工具设计需**预留 AI 智能体接口**，使得大模型能够通过结构化接口自动生成复杂动画，降低创作门槛。

### 1.3 目标用户
- **主要用户**：ACM集训队教练及队长（内容创作者）；
- **次要用户**：集训队队员（学习观看者）；
- **潜在用户**：算法讲师、技术博客作者、开源文档维护者；
- **AI 智能体**：未来可通过自然语言交互生成动画的辅助系统。

## 2. 核心功能需求

| 功能模块 | 详细描述 |
|---------|---------|
| **演示文稿生成** | 使用 Markdown 编写内容，通过简单分页标记自动生成幻灯片；提供预设主题与排版模板，用户无需关心布局细节。 |
| **代码与公式支持** | 支持主流编程语言代码高亮、LaTeX 数学公式渲染、Mermaid 流程图等。 |
| **算法可视化引擎** | 通过 JavaScript 脚本描述动画逻辑（如数组元素交换、指针移动、树节点高亮），内置常用数据结构的可视化 API，支持播放、步进、调速等交互控制。 |
| **组件化嵌入** | 在 Markdown 中通过自定义标签（如 `<ArrayViz />`）插入动画组件，实现内容与演示的无缝集成。 |
| **导出与分享** | 支持导出为独立 HTML（包含完整动画与交互）或 PDF 文档；生成的 HTML 可一键部署至 Web 服务器，通过链接分享。 |
| **AI 智能体接口** | 提供结构化的动画定义格式（如 JSON 步骤描述）和 CLI 工具，允许 AI 生成动画脚本，并通过命令行自动集成到演示文稿中。 |

## 3. 技术架构设计

### 3.1 总体架构

项目采用 **前后端同构（纯前端）** 的架构，所有资源在用户浏览器中运行，无需服务器端支持。但与 Slidev 的关系调整为 **借鉴思想，自主实现核心渲染层**，避免深度绑定，保证未来独立迭代能力。

| 层次 | 职责 | 关键技术 |
|------|------|----------|
| **内容层** | 用户编写 Markdown 文件，包含文本、代码、公式及组件引用 | Markdown |
| **解析与渲染层** | 解析 Markdown，生成幻灯片页面，提供页面导航、主题系统。**借鉴 Slidev 核心设计，自主实现轻量解析器与渲染器**，仅复用必要的语法解析逻辑（如 `---` 分页、代码高亮插件等）。 | 自研 Markdown 解析器（基于 `marked` 或 `unified`）+ Vue 3 |
| **组件层** | 封装算法可视化组件（ArrayViz, TreeViz, GraphViz 等），提供统一的 API 接口 | Vue 3 组件 + TypeScript |
| **动画引擎层** | 负责动画的渲染与控制，包括 Canvas 绘制、补间动画、步骤管理 | Canvas API + Anime.js + 自研播放器 |
| **AI 接口层** | 提供动画步骤的 JSON Schema 定义，以及 CLI 工具，使 AI 能够生成符合规范的动画脚本并注入项目。 | JSON Schema, Commander.js (CLI) |

### 3.2 模块化设计

项目采用模块化开发，核心模块包括：

- **`core/`**：
  - **`markdown-parser/`**：自定义 Markdown 解析器，支持 `---` 分页、代码块识别、组件标签识别。
  - **`renderer/`**：将解析后的 AST 渲染为 Vue 组件树。
  - **`theme/`**：主题系统，支持 CSS 变量和动态切换。
  - **`animation-player/`**：动画播放器核心，管理步骤队列、播放状态、速度控制。
- **`visualizations/`**：各数据结构可视化组件实现，依赖 core 模块进行动画驱动。
- **`utils/`**：通用工具函数，如数据结构操作、坐标计算、颜色管理等。
- **`types/`**：TypeScript 类型定义，确保组件间接口一致性。
- **`cli/`**：命令行工具，用于生成动画脚本、添加新组件、导出文稿等。
- **`ai/`**：AI 交互模块，定义动画步骤的 JSON Schema，提供示例和验证工具。

## 4. 核心技术选型及理由

| 技术组件 | 选型 | 理由 |
|----------|------|------|
| **Markdown 解析** | `unified` + `remark` 生态 + 自定义插件 | 高度可扩展，便于提取分页信息、识别自定义组件标签，且轻量。 |
| **前端框架** | Vue 3 | 轻量、响应式，与 Slidev 同生态但我们可以只取所需，避免打包整个 Slidev。 |
| **构建工具** | Vite | 极快的冷启动与热更新，支持 Vue 单文件组件，开发体验友好。 |
| **动画引擎** | Canvas API + Anime.js | Canvas 渲染性能高，Anime.js 体积小（~17KB），满足算法演示需求。 |
| **播放控制** | 自研 AnimationPlayer | 轻量级步骤管理器，不依赖外部状态库，确保动画逻辑与 UI 解耦。 |
| **AI 交互** | JSON Schema + Commander.js | 提供标准化的动画步骤描述格式，CLI 工具方便 AI 生成后直接集成。 |
| **辅助库** | KaTeX, Shiki, Mermaid | 可独立引入，按需加载，保持轻量。 |
| **包管理** | pnpm | 节省磁盘空间，安装速度快。 |
| **代码规范** | ESLint + Prettier | 统一代码风格，保证代码质量。 |
| **部署** | GitHub Pages / Vercel | 免费、与 Git 仓库联动，推送即部署，满足“分享链接”的需求。 |

## 5. AI 智能体接口设计

### 5.1 设计目标
- 允许大模型通过 **自然语言描述** 生成算法动画。
- 提供 **标准化的动画步骤格式**，AI 可直接输出 JSON 脚本。
- 提供 **CLI 工具**，将生成的脚本快速集成到演示文稿中。

### 5.2 动画步骤 JSON Schema

定义一个通用的步骤描述格式，AI 只需按此格式输出 JSON，工具即可解析并生成动画。

```json
{
  "version": "1.0",
  "type": "array",          // 可选: array, tree, graph
  "initialData": [3, 1, 4, 1, 5],
  "steps": [
    {
      "action": "highlight",
      "target": [0, 2],     // 高亮索引 0 和 2
      "duration": 500
    },
    {
      "action": "swap",
      "target": [0, 1],     // 交换索引 0 和 1
      "duration": 300
    },
    {
      "action": "setValue",
      "target": 3,          // 将索引 3 设为 9
      "value": 9
    }
  ]
}
```

### 5.3 CLI 工具接口

```bash
# 生成动画脚本（AI 输出 JSON 文件）
algoflow generate --type array --description "冒泡排序过程" --output steps.json

# 将动画脚本嵌入到指定的 Markdown 文件
algoflow embed --md slides/sort.md --animation steps.json --tag "<ArrayViz />"

# 预览动画
algoflow preview steps.json
```

### 5.4 AI 集成工作流

1. 用户提供自然语言描述（如“帮我做一个数组冒泡排序的可视化，数组初始为[5,2,9,1]”）。
2. AI（如通过 API 调用）根据内置的 **Schema 和示例** 生成 JSON 步骤文件。
3. 用户运行 `algoflow embed` 将 JSON 注入到对应的 Markdown 幻灯片中。
4. 最终生成包含动画的演示文稿。

## 6. 工程化设计

### 6.1 项目目录结构（示意）

```
algoflow/
├── .github/               # GitHub Actions 配置（CI/CD）
├── .vscode/               # 编辑器推荐配置
├── packages/              # 多包管理（如有需要）
│   ├── core/              # 核心解析与渲染逻辑
│   ├── cli/               # 命令行工具
│   └── visualizations/    # 可视化组件库
├── public/                # 静态资源（图片、视频等）
├── slides/                # 用户编写的 Markdown 笔记（示例）
├── scripts/               # 构建与辅助脚本
├── .eslintrc.js
├── .prettierrc
├── package.json
├── pnpm-workspace.yaml    # pnpm 工作区配置
└── README.md
```

### 6.2 构建与部署

- **开发环境**：运行 `pnpm run dev` 启动 Vite 开发服务器，支持 Markdown 与组件热更新。
- **生产构建**：运行 `pnpm run build` 生成静态网站（HTML + JS + CSS），产物位于 `dist/`。
- **部署**：将 `dist/` 推送到 GitHub Pages 或 Vercel 即可在线访问。
- **导出 PDF**：使用 `pnpm run export` 命令（通过 puppeteer 渲染）生成 PDF，适用于打印或离线阅读。

### 6.3 性能优化策略

| 优化项 | 策略 |
|--------|------|
| **首屏加载** | 代码分割（Vite 默认支持），按需加载可视化组件，避免一次性加载所有动画库。 |
| **动画性能** | 使用 Canvas 而非 DOM 操作，利用 `requestAnimationFrame` 进行帧动画；动画步骤预计算，减少运行期计算量。 |
| **资源体积** | 选用轻量级库（Anime.js 仅 17KB），避免引入大型依赖（如 Three.js 初期不引入）。 |
| **缓存** | 导出 HTML 时资源内联或使用长期缓存策略；静态资源通过 CDN 加速。 |

## 7. 非功能性需求

| 需求 | 指标 |
|------|------|
| **轻量化** | 核心动画引擎（含 Anime.js）打包后增量体积 < 100KB；完整工具包（不含示例）< 1MB。 |
| **性能** | 动画帧率稳定在 60fps（1000 个元素以内）；首屏加载时间 < 3s（4G 网络）。 |
| **兼容性** | 支持最新版 Chrome、Firefox、Edge 浏览器（现代 Web 标准）。 |
| **易用性** | 用户只需掌握 Markdown 基本语法和简单组件标签即可使用；AI 接口提供自然语言生成动画的能力，进一步降低门槛。 |
| **可扩展性** | 通过插件机制或组件注册方式，支持第三方贡献新的可视化组件和动画步骤类型。 |

## 8. 风险与应对策略

| 风险 | 应对措施 |
|------|----------|
| **自研解析器复杂性** | 借鉴 Slidev 的解析逻辑但只取核心部分，利用 `unified` 生态降低开发难度；分阶段实现，先支持基础功能。 |
| **AI 生成的 JSON 可能不符合规范** | 提供 Schema 验证工具（`algoflow validate`），并给出错误提示；训练阶段提供大量示例供 AI 学习。 |
| **动画复杂度超出预期** | 初期限定组件支持的动画类型（数组交换、节点高亮等），复杂动画通过视频补充；后续迭代逐步丰富。 |
| **性能瓶颈** | 采用增量绘制技术（只重绘变化区域），或引入 Web Worker 进行数据处理。 |
| **团队协作代码规范** | 通过 ESLint + Prettier + Husky 强制统一代码风格，确保 Pull Request 质量。 |
